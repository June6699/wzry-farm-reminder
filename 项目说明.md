# 项目说明

这份文档是给开发和学习用的，重点解释：

- 每个文件夹和文件是干什么的
- 前端关键样式和布局是怎么组织的
- 后端的工作逻辑是怎样串起来的
- 页面操作之后，数据是如何流转的

如果你只是想知道“怎么装、怎么跑、怎么用”，优先看 `README.md`。  
如果你想读代码、改代码、继续扩展，优先看这份文档。

## 1. 项目总览

当前项目是一个“本地 Web 应用”：

- Python 负责本地 HTTP 服务、规则读取、SQLite 数据处理
- 浏览器负责显示页面和交互
- SQLite 负责持久化
- JSON 负责配置和规则文件

一句话理解：

`Python 标准库后端 + 本地网页前端 + SQLite 数据存储`

### 3.2 `farm_reminder/`

| 文件 | 作用 | 核心内容 |
|---|---|---|
| `app.py` | 应用启动层 | 建目录、初始化规则、初始化数据库、起 HTTP 服务 |
| `paths.py` | 路径集中管理 | 所有数据目录、规则目录、数据库路径都在这里统一定义 |

| `database.py` | 数据层 | 建表、查表、钱包流水、农场格子状态、偷菜记录、活动点图数据 |
| `server.py` | API 层 | 向前端提供 `/api/bootstrap`、规划、农场编辑、偷菜进出账等接口 |

### 3.3 `web/`

### 3.4 `data/`

|---|---|

| `rules/watering_rules.json` | 浇水规则表 |

## 4. 后端工作逻辑

### 4.1 启动流程

启动链路：

1. `main.py`
2. `farm_reminder.app.run()`
3. `ensure_directories()`
4. `ensure_rule_files()`
5. `Database().initialize()`
6. `ThreadingHTTPServer`
7. 浏览器访问 `http://127.0.0.1:8765/`

### 4.2 配置流程

配置逻辑在 `config_store.py`：

- 先读取 `data/config.json`
- 如果没有文件，就写入默认配置
- 如果字段缺失，就自动补默认值
- 如果旧字段还是 `unlocked_field_count`，会自动转换成 `unlocked_slot_count`

也就是说：

- 前端不需要自己做复杂兼容
- 只要发完整或部分配置对象，后端都会清洗

### 4.3 规则流程

规则逻辑在 `rules.py`：

- 首次运行时，如果 `data/rules/*.json` 不存在，就从 `农场信息.md` 解析
- `RuleStore` 统一缓存：
  - 作物表
  - 浇水规则
  - 游戏机制

规划器和前端都通过这个规则层工作，不直接依赖原始 Markdown。

### 4.4 数据流程

数据逻辑主要在 `database.py`。

当前核心数据表：

- `field_slots`
  - 固定 48 个格子，表示 4 块地 × 每块 12 格
- `slot_states`
  - 当前格子是什么状态，种了什么，什么时候开始，什么时候收获
- `wallet_ledger`
  - 钱包流水
- `activity_logs`
  - 种植、浇水、收获、偷菜这些活动点
- `raid_logs`
  - 偷别人菜 / 被别人偷菜
- `theft_logs`
  - 保留兼容的旧偷菜损失记录

### 4.5 API 流程

主要接口在 `server.py`：

| 接口 | 作用 |
|---|---|
| `/api/bootstrap` | 页面启动时一次性拉全局数据 |
| `/api/settings` | 保存设置 |
| `/api/planner` | 生成规划方案 |
| `/api/plans/apply` | 把规划方案应用到已开格子 |
| `/api/slots/update` | 手动改单格/整块/全部的作物、时间、状态 |
| `/api/slots/start` | 手动确认开始种植 |
| `/api/slots/water` | 登记浇水并缩短剩余时间 |
| `/api/slots/harvest` | 登记收获并写入流水 |
| `/api/slots/clear` | 清空所选范围状态 |
| `/api/raids` | 偷别人菜 / 被偷菜进出账 |

## 5. 前端工作逻辑

### 5.1 页面组织

`index.html` 是一个单页应用骨架。

主要结构：

- `.app-shell`
  - 左侧边栏
  - 右侧主面板
- `.section-panel`
  - 首页
  - 农场
  - 规划
  - 统计
  - 偷菜登记
  - 设置

前端不是多页面跳转，而是：

- 所有主页面都预先放在 HTML 里
- 用 `active` 类切换显示

### 5.2 前端状态

`web/app.js` 中有一个统一的 `state` 对象。

主要分两类：

- 后端数据：
  - `config`
  - `crops`
  - `wateringRules`
  - `fields`
  - `wallet`
  - `ledger`
  - `raids`
  - `events`
  - `farmSummary`
  - `planner`
- 前端 UI 状态：
  - 当前页面
  - 当前规划选中的作物
  - 当前农场编辑范围
  - 当前选中的格子
  - 当前统计筛选

这套状态的目的就是：

- 页面刷新前，尽量保留一部分交互上下文
- 页面重新渲染后，还能还原当前操作位置

### 5.3 前端关键布局

`styles.css` 里有几个关键布局类：

| 样式类 | 作用 |
|---|---|
| `.app-shell` | 总体两栏布局 |
| `.sidebar` | 左侧导航和玩家概览 |
| `.main-shell` | 右侧内容区域 |
| `.summary-grid` | 顶部数据卡片网格 |
| `.two-panel-grid` | 常见左右双栏布局 |
| `.farm-layout` | 农场四块地的 2 × 2 容器 |
| `.slot-grid` | 单块地内部 4 × 3 格子布局 |
| `.slot-card` | 单个格子卡片，使用 `aspect-ratio: 1 / 1` 保证正方形 |
| `.chart-shell` | 统计图容器 |
| `#stats-chart` | SVG 散点图画布 |

### 5.4 前端关键样式思路

当前前端样式方向：

- 暗色控制台风格
- 尽量降低泛黄、玻璃态和“大白卡 AI 仪表盘感”
- 采用较克制的蓝/青色强调色
- 重点区域通过深色面板和浅色文字形成层次

为什么这样做：

- 更贴近你给的控制台风格参考图
- 在 Edge 上比之前的大量模糊和浅色大卡片更稳定
- 更适合承载“表单 + 表格 + 图表 + 农场格子”这种高信息密度界面

## 6. 业务逻辑怎么串起来

### 6.1 规划 -> 应用 -> 编辑

核心流程：

1. 规划页生成方案
2. 选一条方案点击“应用到已开格子”
3. 后端把方案写入 `slot_states`，状态是 `planned`
4. 前端跳到农场页
5. 用户继续改单格、改单块、改全部
6. 用户在游戏里真的种完后，再点“手动开始种植”
7. 状态变成 `planted`
8. 后续登记浇水、登记收获

### 6.2 为什么“已规划”和“已种植”要分开

这是当前项目的一个核心设计：

- `planned`：只是你在工具里把计划布置好了
- `planted`：表示你真的已经去游戏里操作并确认种下

这样做的原因：

- 避免“方案刚应用就自动开始倒计时”的假状态
- 允许用户先布置方案，晚一点再真的种
- 让“计划时间已过，待手动开始”这种状态变得合理

### 6.3 浇水逻辑

当前版本的浇水逻辑不是强行自动推断，而是：

- 允许用户自己决定这次扣多少分钟
- 工具按用户输入去缩短收获时间
- 然后刷新格子剩余时间和统计记录

这样做的好处：

- 能覆盖“用户提前浇水”“忍不住乱浇”的情况
- 不会把所有人都强行绑死在理想策略上

### 6.4 钱包逻辑

钱包不是单独保存一个总余额，而是：

- 所有变动写入 `wallet_ledger`
- 当前余额按流水求和重算

这样好处是：

- 不容易算错
- 能还原每一步怎么来的
- 后面要导出或做图更方便

### 6.5 统计图逻辑

统计图使用 `activity_logs + theft/raid` 转成统一事件流：

- 种植
- 浇水
- 收获
- 被偷菜
- 偷别人菜

然后按：

- 横轴 = 日期
- 纵轴 = 一天中的分钟数

绘制到 SVG 散点图里。

优点：

- 不依赖额外图表库
- 体积小
- 自定义交互比较自由

## 7. 当前代码最重要的几个点

### 7.1 `planner.py`

它负责：

- 防偷窗口判断
- 周末双倍判断
- 自然成熟方案
- 浇水压缩方案
- 找最早防偷方案
- 找最优周末双倍方案
- 找双条件叠加方案

### 7.2 `database.py`

它是整个项目最核心的业务层。

当前主要职责：

- 初始化数据库
- 返回农场格子状态
- 记录钱包流水
- 应用规划
- 手动修改格子
- 开始种植
- 浇水
- 收获
- 记录偷菜进出账
- 返回统计点图数据

### 7.3 `app.js`

它是前端的核心控制器。

你如果后面要继续改前端，优先读这几个函数：

- `applyDashboard`
- `renderAll`
- `renderFarmEditor`
- `renderFarmLayout`
- `renderPlanner`
- `renderStats`
- `bindPlanner`
- `bindFarm`
- `bindRaidForms`

## 9. 当前限制

当前版本还没有做的东西：

- 真正的提醒调度器
- 系统托盘常驻
- 邮件提醒
- 作物 icon 下载
- 打包为 exe
- macOS 版本
- 部署到远程站点

如果以后要做：

- 邮件提醒
- 远程站点
- 多设备同步
- 多用户

那么架构大概率要改成：

- 前后端分离
- API 服务常驻
- 独立静态前端构建
- 更明确的认证和数据同步层

但好处是：  
现在的业务逻辑已经拆得比较清楚了，尤其是：

- 规划逻辑
- 农场状态逻辑
- 钱包流水逻辑
- 活动统计逻辑

这些都能继续复用。
